# はじめに

こんにちは。かろっく@calloc134 です。

SPA + API 構成でアプリケーションを構築する際、CSRF 攻撃への対策は避けて通れないテーマです。
しかし、CSRF 攻撃の防御機構は CORS ポリシー、SameSite 属性、Origin ヘッダ検証など、複数の仕組みが絡み合っており、それぞれがどのように機能するのかを横断的に理解している人は少ないのではないでしょうか。

この記事では、SPA + API 構成における CSRF 攻撃・クロスサイト読み取り攻撃の防御機構について、
前提知識から実践的な対策まで、横断的に解説していきます。

今回は Hono フレームワークをベースに解説を進めていきますが、
基本的な考え方は他のフレームワークでも応用可能です。

# 忙しい人向けの結論

先に結論をお伝えします。以下の対策を実施すれば、CSRF は基本的に防御できます。

- `hono/cors` で正しく CORS ポリシーを設定する
  - ブラウザの同一オリジンポリシー (SOP) による防御機構を有効にする
- `hono/csrf` ミドルウェアを利用して Origin ヘッダ検証を行う
  - simple request と認識されるリクエストに対して耐性がある
- クッキーの SameSite 属性を適切に設定する
  - 完全同一ドメイン・サブドメインの場合: `SameSite=Lax`
  - 完全別ドメインの場合: `SameSite=None` + `Secure` 属性
  - CSRF とは関係ないが、以下の属性は必ず設定すること
    - `HttpOnly` 属性
    - `Secure` 属性
- データを変更する API (副作用あり) と データを取得する API (副作用なし) で メソッドを分離する
  - データを変更する API (副作用あり): unsafe method (POST, PUT, PATCH, DELETE)
  - データを取得する API (副作用なし): safe method (GET, HEAD, OPTIONS, TRACE)

では、これらの対策がなぜ必要なのか、詳しく見ていきましょう。

# 前提知識

CSRF 攻撃・クロスサイト読み取り攻撃に関する前提知識は非常に多岐にわたります。
まずは前提知識を整理していきます。

## CSRF 攻撃とは

CSRF (Cross-Site Request Forgery) 攻撃とは、
攻撃者が用意した攻撃用サイトに被害者がアクセスした際に、
被害者のブラウザが正規サイトに対して意図しないリクエストを送信させ、
結果として**被害者の認証情報で不正な状態変更操作が行われる**攻撃手法です。

今回は、正規 API に対して意図しないリクエストを送信させるケースを想定します。

この攻撃が成立するのは、**認証情報が自動で付与される場合**です。
例えば、クッキーを用いたセッション管理が行われている場合、
クッキーはブラウザによって自動で付与されるため、攻撃が成立する可能性があります。

## クロスサイト読み取り攻撃とは

クロスサイト読み取り攻撃とは、
攻撃者が用意した攻撃用サイトに被害者がアクセスした際に、
被害者のブラウザが正規サイトに対してリクエストを送信し、
攻撃用サイトでそのレスポンス内容を取得することで、
結果として**被害者の認証情報で正規サイトの情報が攻撃者に漏洩する**攻撃手法です。

今回は、正規 API に対してリクエストを送信し、
そのレスポンス内容を攻撃者に漏洩させるケースを想定します。

:::message
クロスサイト読み取り攻撃が発生する前提で記述していますが、
実際は後述の同一オリジンポリシー (SOP) の仕組み上、多くの場合で防止されます。
:::

:::message
XS-Leak (Cross-Site Leak) とは区別します。
同一オリジンポリシー (SOP) によってデータ読み取りが防止される場合でも、
レスポンスの有無やレスポンス時間などの副次的情報を利用して
間接的に情報を漏洩させる攻撃手法があります。
今回は XS-Leak は考慮しません。
:::

## HTTP メソッドの使い分け

HTTP メソッドは、その性質によって大きく 2 つに分類されます。

### unsafe method

unsafe method には POST, PUT, PATCH, DELETE が該当します。
このメソッドで呼び出される API は、
**データベースの更新など副作用が発生する操作を行うもの**と期待されます。
予測される脅威は **CSRF 攻撃**です。

### safe method

safe method には GET, HEAD, OPTIONS, TRACE が該当します。
このメソッドで呼び出される API は、
**データの取得など副作用が発生しない操作を行うもの**と期待されます。
予測される脅威は**クロスサイト読み取り攻撃**です。

API で行う処理に応じて適切なメソッドを選択することが重要です。
詳細は後述しますが、この使い分けは CSRF 攻撃の防御において非常に重要な役割を果たします。

## オリジンとは

オリジンとは、**スキーム + ホスト + ポート** の組み合わせです。
ポートが省略された場合は、スキームから導出されます。

具体例を見てみましょう。

| 比較対象                                            | 同一オリジン？ |
| --------------------------------------------------- | -------------- |
| `https://example.com` と `https://example.com`      | ✅ 同一        |
| `https://example.com` と `https://api.example.com`  | ❌ 別オリジン  |
| `https://example.com` と `https://example.org`      | ❌ 別オリジン  |
| `https://example.com` と `http://example.com`       | ❌ 別オリジン  |
| `https://example.com:443` と `https://example.com`  | ✅ 同一        |
| `https://example.com:8080` と `https://example.com` | ❌ 別オリジン  |

## サイトとは

サイトとは、**スキーム + registrable ドメイン**の組み合わせです。

registrable ドメインとは、
第三者が自由に登録できるサフィックス (public suffix) と
**その一つ左側のラベル**を組み合わせたものです。

### public suffix の種類

public suffix には大きく 2 種類あります。

**A: ICANN ドメイン**
トップレベルドメイン (TLD) であり、レジストラが管理するドメインです。

- `(ラベル).com`
- `(ラベル).co.uk`

**B: 非 ICANN ドメイン (PRIVATE ドメイン)**
TLD ではないが、単一サービスが利用者にサブドメインを提供するドメインです。

- `(ラベル).github.io`
- `(ラベル).workers.dev`

### 具体例

| 比較対象                                                             | 同一サイト？ |
| -------------------------------------------------------------------- | ------------ |
| `https://example.com` と `https://api.example.com`                   | ✅ 同一      |
| `https://example.com` と `https://example.org`                       | ❌ 別サイト  |
| `https://app1.user1.workers.dev` と `https://app2.user1.workers.dev` | ✅ 同一      |
| `https://app.user1.workers.dev` と `https://app.user2.workers.dev`   | ❌ 別サイト  |

「一つ左側のラベル」が同じかどうかがポイントです。

## 同一オリジンポリシー (SOP) と CORS ポリシー

同一オリジンポリシー (SOP) と CORS ポリシーは、
**クロスオリジンリクエストにおけるブラウザの挙動を制御する仕組み**です。

### 同一オリジンポリシー (SOP)

ブラウザには同一オリジンポリシー (SOP) というポリシーが存在し、
**別オリジンのリソースにアクセスすることを制限**します。

クロスオリジンリクエストを行った場合の同一オリジンポリシーの挙動は、
メソッドによって異なります。

**unsafe method の場合**

- **リクエストの送信自体の可否**を制御します
- API が呼び出された際に副作用が発生する可能性があるため、
  リクエストの送信自体を制御・制限し、CSRF 攻撃を防止します

:::message alert
unsafe method の場合でも、特定の条件 (後述の simple request) を満たす場合は、
safe method と同じ挙動となることがあります。
:::

**safe method の場合**

- **レスポンスの閲覧可否**を制御します
- API が呼び出された際に副作用が発生しないため、
  リクエストの送信自体は許可しますが、
  レスポンスの閲覧可否を制御・制限し、クロスサイト読み取り攻撃を防止します

### CORS ポリシー

CORS ポリシーは、この SOP を**緩和するための仕組み**です。
`Access-Control-*` ヘッダを用いて、
リクエスト先のオリジンが許可リストに含まれている場合に限り、
リクエスト送信・レスポンス閲覧を例外的に許可します。

### 主な CORS ヘッダ

| ヘッダ                             | 説明                                                                                  |
| ---------------------------------- | ------------------------------------------------------------------------------------- |
| `Access-Control-Allow-Origin`      | クロスオリジンのリクエスト送信・レスポンス閲覧を許可するオリジンを指定                |
| `Access-Control-Allow-Credentials` | クロスオリジンのリクエストにおいてクレデンシャル (Cookie 等) を含めることを許可するか |

**重要な制約**

`Access-Control-Allow-Origin` にワイルドカード `*` を指定している場合、
`Access-Control-Allow-Credentials` を `true` に設定できないという制約があります。

これにより、CORS ポリシーの設定ミスによる CSRF 攻撃・クロスサイト読み取り攻撃を防止できます。

### CORS ポリシーが有効な場合の挙動

**unsafe method かつ simple request でない場合**

1. ブラウザの JavaScript が fetch API を用いて unsafe method のリクエストを送信しようとする
2. ブラウザが CORS ポリシーを確認するための**プリフライトリクエスト**を送信 (OPTIONS メソッド)
3. サーバーが CORS ポリシーを含むレスポンスを返す
4. ブラウザがレスポンスを確認し、CORS ポリシーに従って送信可否をチェック
5. 送信が許可されていれば本リクエストを送信、許可されていなければ送信をブロック

これにより、リクエストの送信自体を制御・制限し、CSRF 攻撃を防止できます。

**safe method の場合**

1. JavaScript が fetch API を用いて safe method のリクエストを送信しようとする
2. ブラウザがリクエストを送信 (**リクエストの送信自体は制限されない**)
3. サーバーが CORS ポリシーを含むレスポンスを返す
4. ブラウザがレスポンスを確認し、CORS ポリシーに従って閲覧可否をチェック
5. 閲覧が許可されていればレスポンスが JavaScript に渡される、許可されていなければエラー

これにより、レスポンスの閲覧可否を制御・制限し、クロスサイト読み取り攻撃を防止できます。

:::message alert
挙動の流れを見るとわかるように、
CORS ポリシーはレスポンスの閲覧可否のみを制御・制限するため、
**API へのアクセス自体は防止されません**。

したがって、副作用の発生する API が safe method で実装されている場合、
CSRF 攻撃が成立してしまう可能性があります。
:::

### simple request とは (例外)

unsafe method であっても、以下の条件をすべて満たす場合は safe method と同じ挙動になります。

- POST メソッドである
- リクエストヘッダが以下のもののみで構成されている
  - Accept
  - Accept-Language
  - Content-Language
  - Content-Type (ただし以下のいずれかの値に限る)
    - `application/x-www-form-urlencoded`
    - `multipart/form-data`
    - `text/plain`

この条件を満たすリクエストを **simple request** と呼びます。

この仕組みはブラウザが互換性のために備えているレガシーな仕組みであり、
セキュリティ上の観点からは望ましくありません。
CSRF 攻撃に悪用されるリスクがあります。

したがって、開発者は要件に不要な場合、
これらの条件に該当するリクエストを API で受け入れないようにすることが望ましいです。

## SameSite 属性とは

SameSite 属性は、クッキーの属性の一つであり、
**クロスサイトリクエストにおけるクッキーの送信挙動を制御**します。

| 設定値   | 説明                                                 |
| -------- | ---------------------------------------------------- |
| `Strict` | クロスサイトリクエストではクッキーを送信しない       |
| `Lax`    | 一部のクロスサイトリクエストでのみクッキーを送信する |
| `None`   | クロスサイトリクエストでもクッキーを送信する         |

### Lax の詳細

`Lax` では、safe method (GET, HEAD, OPTIONS, TRACE) かつ
特定の条件 (トップレベルナビゲーション等) に当てはまる場合のみクッキーを送信します。

今回想定するクロスサイト読み取り攻撃は fetch API を用いたリクエスト送信を想定しているため、
この条件に当てはまりません。
したがって、`Lax` を設定していればクロスサイトリクエストによるクッキー送信は基本的に防止されます。

:::message
SameSite 属性を設定しない場合のデフォルト値は `Lax` になりますが、
明示的に指定した場合と比べて挙動が異なる場合があるので注意してください。
今回は明示的に `Lax` を設定した場合を想定します。
:::

### None の注意点

`None` を設定する場合、**Secure 属性も設定する必要があります**。

# 想定する状況

本記事では、以下の 3 パターンについて考察します。

| パターン         | 具体例                                 | オリジン | サイト |
| ---------------- | -------------------------------------- | -------- | ------ |
| 完全同一ドメイン | `app.example.com` と `app.example.com` | 同一     | 同一   |
| サブドメイン     | `api.example.com` と `app.example.com` | **別**   | 同一   |
| 完全別ドメイン   | `api.example.org` と `app.example.com` | **別**   | **別** |

# 想定するケースの前提

CSRF 攻撃・クロスサイト読み取り攻撃の防御機構を正常に作用させるため、
開発者は前提として以下の設計を守るものとします。

## API のセッション管理は httponly クッキーを利用する

- クッキーが存在していないと API 呼び出しが行えないものとする
- fetch API において `credentials: include` を設定した呼び出しを想定する

## メソッドの分離

- データを変更する API (副作用あり): unsafe method を用いる (POST, PUT, PATCH, DELETE)
- データを取得する API (副作用なし): safe method を用いる (GET, HEAD, OPTIONS, TRACE)

メソッド分離の重要性は前述の通りです。
API 設計の段階でしっかりと考慮しましょう。

## クッキーの SameSite 属性

| ケース                         | SameSite 属性              |
| ------------------------------ | -------------------------- |
| 完全同一ドメイン・サブドメイン | `SameSite=Lax`             |
| 完全別ドメイン                 | `SameSite=None` + `Secure` |

# 根本的解決は Origin ヘッダ検証

CSRF 攻撃の防御の一番根本的な解決策は **Origin ヘッダ検証**です。

Origin ヘッダを検証していれば、
完全同一ドメイン・サブドメイン・完全別ドメインのいずれの場合でも
CSRF 攻撃を防御できます。

なお、CSRF 攻撃は主に unsafe method が前提となる攻撃なので、
以後のヘッダ検証対策は unsafe method (POST, PUT, PATCH, DELETE) の場合を想定します。
逆に safe method (GET, HEAD, OPTIONS, TRACE) の場合は、これらの検証を省略しても問題ありません。

## Origin ヘッダ検証の流れ

1. Origin ヘッダの値が予期されるオリジンと一致するかを確認
2. 一致すれば許可、一致しなければ拒否

非常にシンプルですが、これが最も確実な防御策です。

## 本記事の目的

しかし今回は、**Origin ヘッダ検証を怠った場合**において、
Cookie の SameSite 属性や CORS ポリシーによる
CSRF 攻撃の副次的防御の影響について考察することを目的とします。

:::details 余談: Sec-Fetch-Site ヘッダ検証について

最近のブラウザでは `Sec-Fetch-Site` ヘッダが送信されます。
Origin ヘッダ検証の補助として `Sec-Fetch-Site` ヘッダ検証を利用することで、
Origin ヘッダ検証よりもパフォーマンスの良い CSRF 防御を実現できる可能性があります。

`Sec-Fetch-Site` ヘッダを検証するときは以下のような実装となります。

**1. Sec-Fetch-Site ヘッダが存在する場合**

前提として、`Sec-Fetch-Site` ヘッダの値が `none` の場合は一括で拒否します。
その場合、ヘッダの値は以下の 3 種類となります。

- `same-origin`
- `same-site`
- `cross-site`

| ケース           | 正規リクエスト | 判定ロジック                                                                    |
| ---------------- | -------------- | ------------------------------------------------------------------------------- |
| 完全同一ドメイン | `same-origin`  | `same-origin` → 許可、それ以外 → 拒否                                           |
| サブドメイン     | `same-site`    | `same-origin` または `same-site` → Origin ヘッダ検証に移行、`cross-site` → 拒否 |
| 完全別ドメイン   | `cross-site`   | すべて Origin ヘッダ検証に移行                                                  |

**2. Sec-Fetch-Site ヘッダが存在しない場合**

古いブラウザ等では送信されないことがあるため、
Origin ヘッダ検証にフォールバックします。

`Sec-Fetch-Site` ヘッダ検証を行っていても、Origin ヘッダ検証との併用が前提となります。
:::

# Origin ヘッダ検証を怠った場合の CSRF 攻撃の成立可否

ここからは、Origin ヘッダ検証を怠った場合に、
CSRF 攻撃・クロスサイト読み取り攻撃が成立するかどうかを考察します。

表の凡例は以下の通りです。

- **横軸**: 被害者に踏ませるための攻撃者のページのドメイン
- **縦軸**: 本来の通信元である正規の SPA ページの構成
- **A**: 適切な SameSite 属性を設定した際の、クッキーの送信挙動による CSRF 耐性
- **B**: 適切な CORS ポリシーを設定した際の、同一オリジンポリシー (SOP) による CSRF 耐性

## α. fetch API を用いた unsafe method による CSRF 攻撃

攻撃者ページからの fetch API を用いた unsafe method によるリクエスト送信が成功すること、
つまり**送信自体がブロックされないこと**が攻撃成功の条件です。

それぞれの条件において、CSRF 攻撃を防御する防御機構が働くかを以下の表に示します。

| 正規 SPA の構成＼攻撃者ページ                                     | サブドメイン (別サブドメイン) | 完全別ドメイン |
| ----------------------------------------------------------------- | ----------------------------- | -------------- |
| 完全同一ドメイン `SameSite=Lax`                                   | B                             | A, B           |
| サブドメイン `SameSite=Lax` `Access-Control-Allow-Credentials`    | B                             | A, B           |
| 完全別ドメイン `SameSite=None` `Access-Control-Allow-Credentials` | B                             | B              |

### 考察

この表から以下のことがわかります。

- **すべての例において、同一オリジンポリシー (SOP) による防御機構 (CORS ポリシー) が働く**
- 正規の SPA ページが完全同一ドメイン・サブドメインであり、かつ攻撃者ページが完全別ドメインである場合には、クッキーの SameSite 属性による防御機構も働く

したがって、**Origin ヘッダ検証を怠った場合でも、この攻撃パターンでは CSRF 攻撃は成立しない**ことがわかります。

## β. simple request を用いた CSRF 攻撃

`text/plain` 等の simple request と認識されるコンテンツタイプを用いて、
POST で JSON データを無理やり送信させるタイプの CSRF 攻撃です。

攻撃者ページからの fetch API もしくはフォーム送信を用いた POST によるリクエスト送信が成功すること、
つまり**送信自体がブロックされないこと**が攻撃成功の条件です。

それぞれの条件において、CSRF 攻撃を防御する防御機構が働くかを以下の表に示します。

| 正規 SPA の構成＼攻撃者ページ                                     | サブドメイン (別サブドメイン) | 完全別ドメイン |
| ----------------------------------------------------------------- | ----------------------------- | -------------- |
| 完全同一ドメイン `SameSite=Lax`                                   | **なし**                      | A              |
| サブドメイン `SameSite=Lax` `Access-Control-Allow-Credentials`    | **なし**                      | A              |
| 完全別ドメイン `SameSite=None` `Access-Control-Allow-Credentials` | **なし**                      | **なし**       |

### 考察

この表から以下のことがわかります。

- **「なし」と示されている部分では、CSRF 攻撃を防御する防御機構が働かない**
- つまり、**CSRF 攻撃に脆弱**である
- 正規の SPA ページが完全同一ドメイン・サブドメインであり、かつ攻撃者ページが完全別ドメインである場合には、クッキーの SameSite 属性による防御機構が働く

### この攻撃方法の特徴

この攻撃方法の一番の特徴は、
**JSON データのみを受け付けるべきである API に対して、
simple request の範疇に収まる条件のリクエストを送信できてしまう点**が問題です。

したがって、以下の条件に該当するものは、そもそも API で受け入れないことが望ましいです。

- POST メソッドである
- Content-Type が以下のいずれか
  - `application/x-www-form-urlencoded`
  - `multipart/form-data`
  - `text/plain`

### フレームワークの実装に関する注意

API フレームワークの実装によっては、
JSON を受け取る際に本当にコンテンツタイプが `application/json` であるかを検証するものもあります。
しかし、そのような実装になっていない場合も多いです。

例えば、Hono の `c.req.json()` メソッドは**コンテンツタイプを検証しません**。
開発者がしっかり自衛を行うことが大事です。

### Hono での対策

Hono の場合は `hono/csrf` ミドルウェアを提供しています。
simple request と認識されるリクエストに対して、
`Origin` ヘッダ検証 (+ `Sec-Fetch-Site` ヘッダ検証) を行うことで、
実質的にこのタイプの CSRF 攻撃を防止できます。

このような設計により、最小限のパフォーマンス低下で
現実的な CSRF 攻撃の脅威を防止しています。

:::details 余談: 他の攻撃手段について

simple request を送信する手段としては、以下が考えられます。

**フォーム送信**
`<form>` タグを用いてフォームを作成し、`submit()` メソッドを呼び出す、
もしくはボタンをクリックさせることでリクエスト送信。

**fetch API**
fetch API を用いて `text/plain` 等の simple request と認識される
コンテンツタイプを用いたリクエスト送信。

また、`text/plain` 以外にも、API 側が受け付ける場合は
`multipart/form-data` や `application/x-www-form-urlencoded` の悪用も同様に考えられます。
:::

## γ. fetch API を用いた safe method によるクロスサイト読み取り攻撃

攻撃者ページからの fetch API を用いた safe method によるリクエスト送信が成功し、
**レスポンスの内容が攻撃者ページの JavaScript に渡されること**が攻撃成功の条件です。

:::message
この場合は safe method なので、CORS の防衛ラインの軸足が
「API を呼び出さない」から「API 呼び出しは許容するがそのデータは閲覧できない」に変化することに注意してください。
:::

それぞれの条件において、クロスサイト読み取り攻撃を防御する防御機構が働くかを以下の表に示します。

| 正規 SPA の構成＼攻撃者ページ                                     | サブドメイン (別サブドメイン) | 完全別ドメイン |
| ----------------------------------------------------------------- | ----------------------------- | -------------- |
| 完全同一ドメイン `SameSite=Lax`                                   | B                             | A, B           |
| サブドメイン `SameSite=Lax` `Access-Control-Allow-Credentials`    | B                             | A, B           |
| 完全別ドメイン `SameSite=None` `Access-Control-Allow-Credentials` | B                             | B              |

### 考察

この表から以下のことがわかります。

- **すべての例において、同一オリジンポリシー (SOP) による防御機構 (CORS ポリシー) が働く**
- 正規の SPA ページが完全同一ドメイン・サブドメインであり、かつ攻撃者ページが完全別ドメインである場合には、クッキーの SameSite 属性による防御機構も働く

したがって、**Origin ヘッダ検証を怠った場合でも、クロスサイト読み取り攻撃は成立しない**ことがわかります。

:::message alert
**重要**: 副作用の発生する API は safe method で実装しないことが重要です。

safe method において、同一オリジンポリシー (SOP) の防御機構は
レスポンスの取得に関しては制限しますが、**リクエストの送信自体は制限しません**。

そのため、safe method で副作用を発生させる API が存在している場合、
CSRF 攻撃が成立してしまう可能性があります。
:::

:::details 余談: CORS 設定ミスによる脆弱性

`Access-Control-Allow-Origin` にワイルドカード `*` を設定し、かつ
`Access-Control-Allow-Credentials` ヘッダが `true` に設定されている場合は、
ブラウザはレスポンス内容を JavaScript に渡さずエラーにするのは前述の通りです。

ただし、`Access-Control-Allow-Origin` にワイルドカード `*` を設定してしまっており、かつ
クッキー情報なしでアクセスできる API が存在した場合は、
クロスサイト読み取り攻撃が成立してしまう可能性があります。

そもそも、そのような API 設計は今回の前提条件に反するため、ここでは考慮しません。
:::

:::details 余談: Origin ヘッダが付属しないケース

Origin ヘッダは、同一オリジン (= 完全同一ドメイン) で GET, HEAD のみ付属しないという特徴があります。

これは、完全同一ドメインかつ safe method の場合、
CSRF 攻撃の危険性が極めて少なく、心配する必要がないと判断されるからです。

したがって、同一オリジンかつ safe method の場合には
Origin ヘッダ検証を省略することができます。
:::

# 結論

本記事の考察をまとめます。

## α. fetch API を用いた unsafe method による CSRF 攻撃

**Origin ヘッダ検証を怠った場合でも、CSRF 攻撃が成立しないことが多い。**

CORS ポリシーによる同一オリジンポリシー (SOP) の防御機構が働くためです。

## β. simple request を用いた CSRF 攻撃

**Origin ヘッダ検証を怠った場合に、CSRF 攻撃が成立する可能性がある。**

したがって、開発者は以下のいずれかの対策を行うことが望ましいです。

- API の要件に不要な場合は、simple request と認識されるリクエストを API で受け入れないようにする
- Hono の `hono/csrf` ミドルウェアを利用して、simple request と認識されるリクエストに対して Origin ヘッダ検証を行う

## γ. fetch API を用いた safe method によるクロスサイト読み取り攻撃

**Origin ヘッダ検証を怠った場合でも、クロスサイト読み取り攻撃が成立しないことが多い。**

CORS ポリシーによる同一オリジンポリシー (SOP) の防御機構が働くためです。

## 総括

以上より、Origin ヘッダ検証を怠った場合においても、
**基本的には適切な CORS ポリシーを設定した際の同一オリジンポリシー (SOP) による CSRF 耐性**が働くことで、
CSRF 攻撃・クロスサイト読み取り攻撃を防御できる場合が多いことがわかります。

SameSite 属性による CSRF 耐性も副次的に働く場合がありますが、
同一オリジンポリシーよりは防御効果が限定的です。

どちらの場合も、**β の攻撃方法 (simple request を用いた CSRF 攻撃) に対しては防御効果がない場合があります**。
その部分のみ `hono/csrf` ミドルウェアで狙い撃ちの防御を行うことが望ましいです。

## 実践的な対策まとめ

基本的には、最初に述べた通り以下の対策を実施すれば CSRF は基本的に防御できます。

1. `hono/cors` で正しく CORS ポリシーを設定する
2. クッキーの SameSite 属性を適切に設定する
3. データを変更する API (副作用あり) と データを取得する API (副作用なし) で メソッドを分離する
4. `hono/csrf` ミドルウェアを利用して Origin ヘッダ検証を行う

さらに安全を期すのであれば、`hono/csrf` ミドルウェアの代わりに以下の対策を追加することも検討できます。

- すべてのリクエストに対して Origin ヘッダ検証を行う
  - ただし、同一ドメイン = 同一オリジンかつ GET/HEAD の場合は Origin ヘッダが付属しないため、例外的に検証をスキップする
- コンテンツタイプを `application/json` のみ受け入れる

ただし、この場合は独自実装になるため注意が必要です。

# おわりに

この記事では、SPA + API 構成における CSRF 攻撃の防御機構について、
CORS ポリシー、SameSite 属性、Origin ヘッダ検証などの仕組みを横断的に解説しました。

CSRF 攻撃の防御は、単一の対策で完璧に防げるものではありません。
複数の防御機構を組み合わせることで、より堅牢なセキュリティを実現できます。

特に重要なのは、**API 設計の段階でメソッドを適切に分離すること**です。
副作用のある操作には unsafe method を、副作用のない操作には safe method を使用することで、
ブラウザの同一オリジンポリシーによる保護を最大限に活用できます。

この記事が、皆さんの SPA + API 構成におけるセキュリティ対策の一助となれば幸いです。
