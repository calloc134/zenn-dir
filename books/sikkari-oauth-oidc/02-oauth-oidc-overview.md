## OAuth とは

OAuth は、**リソースの所有者が、リソースへのアクセス権を第三者に与える仕組み**です。

![](/images/02-oauth-oidc-overview/2025-12-01-20-06-49.png)

### 具体例：写真印刷サービス

例えば、自分の写真を Google Photos のようなオンラインアルバムサービスで管理しているとします。
このアルバムサービスとは別の印刷サービスに、自分の写真を印刷してもらいたい場合を考えてみましょう。

従来であれば、印刷サービスに Google のパスワードを教え、自分の代わりに写真にアクセスしてもらう必要がありました。
しかし、それは**意図した範囲を超えたアクセスを許可してしまう危険な行為**です。
写真以外にも、メールや連絡先など、他の個人情報にまでアクセスされてしまう可能性があります。

OAuth を使えば、**「印刷サービスにだけ、特別に、自分の写真を見る権限を与える」** ということが実現できます。

![](/images/02-oauth-oidc-overview/2025-12-01-19-40-00.png)

具体的には、
**アクセストークン**と呼ばれる「リソースにアクセスできる鍵」を利用することで、
リソースへのアクセスが可能になります。

自分の写真を見る権限のみを持つアクセストークンを印刷サービスに渡すことで、印刷サービスは自分の写真を見ることができるようになります。

![](/images/02-oauth-oidc-overview/2025-12-01-19-41-18.png)

## OIDC とは

OIDC（OpenID Connect）は、
**外部サービスのユーザ情報を用いて連携を行うための仕組み**です。

![](/images/02-oauth-oidc-overview/2025-12-27-19-02-05.png)

外部ログイン、ソーシャルログイン、という言葉が馴染み深いかもしれません。
外部サービスのユーザ情報を用いて連携を行う仕組みは、一般的に **ID 連携** と呼ばれます。

### 具体例：Google アカウントでのログイン

例えば、Zenn のようなサービスに、Google アカウント連携を用いて ログインしたい場合を考えてみましょう。
従来手法で無理やり実現しようとすると、
Zenn に Google アカウントの ID とパスワードを教え、Zenn が Google に代わってログインする必要がありました。
しかし、それは**ID とパスワードを第三者に教える危険な行為**です。

OIDC を使えば、**「Zenn が Google アカウントの所有者であることを確認できる」** ということが実現できます。

![](/images/02-oauth-oidc-overview/2025-12-27-19-11-55.png)

「Google でログイン」ボタンをクリックすると、Google のログイン画面が表示されます。
Google アカウントでログインすると、Zenn に戻り、ログインが完了します。

このとき Zenn は内部で Google から
**「このユーザは確かに Google アカウントを持っており、ログインに成功した」**
という証明書を受け取っています。
この証明書をもとに、
Zenn はユーザが正当な Google アカウントの所有者であることを確認しています。

つまり、Zenn というサービスが、
Google という外部サービスのユーザ情報を利用して
連携・ログインを実現しているわけです。

具体的には、
**ID トークン**と呼ばれる認証結果の証明書のようなトークンを利用することで、
ログインが実現されています。

## 目的の違い

OAuth と OIDC は、**目的が全く異なるユースケース**に対応しています。

| 項目                       | OAuth                                          | OIDC                                   |
| -------------------------- | ---------------------------------------------- | -------------------------------------- |
| **目的**                   | リソースに対する認可                           | 認証結果の証明トークンを用いた ID 連携 |
| **主な用途**               | 第三者サービスにリソースへのアクセス権を与える | 第三者サービスにログインする           |
| **取得するもの**           | アクセストークン                               | ID トークン（+ アクセストークン）      |
| **トークンを検証する主体** | リソースを提供するサービス（リソースサーバ）   | ユーザがログインする第三者サービス自体 |

一般的に、OAuth は 認可のプロトコル、
OIDC は 認証のプロトコル として表現されることが多いです。

しかし、本書ではより本質的に理解するために、
**OAuth は「リソース認可のプロトコル」**、
**OIDC は「ID 連携のプロトコル」** と表現します。

![](/images/02-oauth-oidc-overview/2025-12-27-19-16-20.png)

:::message alert
**OAuth を ID 連携・外部ログインに利用しないこと！**

OAuth はリソースを認可するための仕組みであり、ログイン用途に設計されたものではありません。
OAuth をログイン用途に転用すると意図されない使われ方となり、セキュリティ上の問題が発生する可能性があります。

詳細は後の章で解説しますが、ログインが必要な場合は OIDC を使用しましょう。
「餅は餅屋」です。
:::

## しかし技術的には「追加レイヤー」

目的は異なりますが、OIDC は技術的には **OAuth の上に構築された追加レイヤー**です。
OIDC は OAuth をベースにしており、
OAuth のフローの中で ID トークンを取得する仕組みを提供しています。

![](/images/02-oauth-oidc-overview/2025-12-27-19-19-44.png)

OIDC の仕様の多くは OAuth の仕組みを流用しており、
両者を明確に切り分けることが難しい部分もあります。
例えば、OIDC の `/userinfo` エンドポイントはアクセストークンを使ってプロフィール情報を取得するため、
OAuth の認可の仕組みを利用していると言えます。

![](/images/02-oauth-oidc-overview/2025-12-01-20-15-20.png)

また、OIDC 側で発祥した仕様が OAuth 側に逆輸入されることもあります。

したがって、**OIDC を理解するためには、まず OAuth の理解が不可欠**です。

本書では、まず OAuth をしっかりと理解してから OIDC の解説に進む構成を採用しています。
では、次の章から OAuth の基礎を学んでいきましょう。

![](/images/02-oauth-oidc-overview/2025-12-01-20-16-06.png)
