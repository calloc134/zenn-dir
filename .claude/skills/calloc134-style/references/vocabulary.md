リライト時に「この語をどう置き換えるか」「ここにどんなつなぎを入れるか」を素早く決めるための辞書的リファレンスです。grep 的に使ってください。

## 1. 接続詞・つなぎ語(本論)

すべて**文頭で読点付き**で使うのが原則。

### 誘導(節・段落の始まり)

| 語 | 使い所 | 例 |
|---|---|---|
| では、 | 節・大段落の入口、話題の切り替え | では、認可コードフローについて見ていきましょう。 |
| ここで、 | 直前の話題に立脚して次に進む | ここで、PKCE の役割について整理しておきましょう。 |
| ここでは、 | 範囲限定の宣言 | ここでは簡単のため、複数トランザクションの話は割愛します。 |
| まず | 順序の 1 番目 | まず、認可リクエストから見ていきます。 |
| まずは | 「最初は手加減して」のニュアンス | まずは Confidential Client を前提に解説します。 |
| 次に | 順序の 2 番目以降 | 次に、トークンレスポンスを見てみましょう。 |
| 最後に | 順序の最後 | 最後に、まとめに入ります。 |

### 言い換え

| 語 | 使い所 |
|---|---|
| つまり、 | 直前を端的に言い直す |
| 要するに、 | やや砕けた言い直し(本論でも使うが頻度は低め) |
| 言い換えると、 | 別の角度からの言い直し |
| 具体的には、 | 抽象 → 具体への橋渡し |

### 補足

| 語 | 使い所 |
|---|---|
| なお、 | 本筋を妨げない注意・条件 |
| ちなみに、 | やや軽い補足 |
| また、 | 並列追加 |
| 更に、 | 強い追加・程度の上乗せ |

### 対比

| 語 | 使い所 |
|---|---|
| しかし、 | 直接的な逆接 |
| 一方、 | 並べ対比 |
| その一方で、 | やや丁寧な並べ対比 |
| 対して、 | 「A は X。対して B は Y。」 |

### 結論導入

| 語 | 使い所 |
|---|---|
| したがって、 | 論理的帰結 |
| そのため、 | 因果 |
| これにより、 | 結果の機構的説明 |

### メタ言及

| 語 | 使い所 |
|---|---|
| 余談ですが、 | 文中での余談導入 |
| 余談として、 | 直前の説明への補足余談 |
| 更に余談ですが、 | 余談の追加 |
| 余談： | `##` 見出しでの余談節 |

## 2. 文末表現

### 本論で使う(柔らかい敬体)

| 表現 | ニュアンス |
|---|---|
| 〜です。 | 基本の断定 |
| 〜となります。 | 結果の宣言 |
| 〜と言えます。 | 柔らかい断定 |
| 〜と考えられます。 | 推論的断定 |
| 〜と思います。 | 主観の表明 |
| 〜になります。 | 状態変化の結果 |
| 〜ます。 | 動詞の基本敬体 |
| 〜していきます。 | これから順に進める宣言 |
| 〜することができます。 | 可能表現(冗長だが筆者は使う) |
| 〜していきましょう。 | 読者を巻き込む誘導 |
| 〜してみましょう。 | やや軽い誘導 |
| 〜してみてください。 | 柔らかい指示 |
| 〜と良いでしょう。 | 推奨 |
| 〜することをおすすめします。 | 推奨(やや強い) |
| 〜のではないでしょうか？ | 推測の問いかけ |
| 〜のでしょうか？ | 節入口の問いかけ |

### イントロ・アウトロでだけ使う

| 表現 | 例 |
|---|---|
| 〜ましょう！ | 「進んでいきましょう！」 |
| 〜ですよね？ | 「ありますよね？」 |
| 〜してみてください！ | 励まし型の指示 |
| 〜お楽しみに〜 | 結びの予告 |
| 〜ですわ / 〜だわ などの女性語 | 使わない |

### 本論で使わない

- 〜だ。/〜である。 (常体)
- 〜じゃん。/〜だね。 (タメ口)
- 〜だぜ。/〜だぞ。 (男性語)
- 〜のだ。 (押し付け感が強い)

## 3. 記号の使い分け

| 記号 | 本論 | イントロ/アウトロ | 用法 |
|---|---|---|---|
| `！` | ❌ (避ける) | ⭕ (`！！！` まで OK) | 強調・感情 |
| `？` | △ (疑問だけ OK) | ⭕ (自信なしマークもOK) | 疑問・問いかけ |
| `・・・` | ❌ | ⭕ | 口ごもり・転調(中黒3つ、`…` ではない) |
| `〜` | △ (範囲記号のみ) | ⭕ (口語の流し OK) | 結びの軽さ |
| `「〜」` | ⭕ | ⭕ | 比喩・新出概念・俗称・セリフ |
| `『〜』` | △ (まれ) | △ | 鉤括弧の入れ子 |
| `()` | ⭕ | ⭕ | 英訳・略称・補足 |
| `（）` | ⭕ | ⭕ | 同上、半全混在を気にしない |
| `**〇〇**` | ⭕ (多用) | ⭕ | 単語・短句単位の太字 |
| `` `xxx` `` | ⭕ (多用) | ⭕ | 識別子・コード片 |

## 4. 比喩のストック

新概念を導入するときに当てる「日常物の比喩」候補。**鉤括弧で括る**のがお約束。

| 抽象概念 | 候補比喩 |
|---|---|
| トークン全般 | 「鍵」「証明書」「引換券」「チケット」「許可証」 |
| 短命トークン | 「引換券」「使い捨て」 |
| 長命トークン | 「身分証」「証明書」 |
| サーバ役割 | 「保管庫」「窓口」「受付」「認可局」 |
| ID トークン | 「証明トークン」「認証結果の証明書」 |
| クライアント | 「第三者」「代理人」「使者」 |
| 状態のスナップショット | 「写し」「コピー」 |
| キュー・タスク | 「行列」「順番待ち」 |
| 優先度・レーン | 「車線」「レーン」 |
| 仮想 DOM の片方 | 「パラレルワールドの自分」 |
| インデックス | 「目次」「索引」 |
| MVCC の物理タプル | 「履歴のスナップショット」 |

導入の構文:
- 「〇〇は、〜のための『□□』です。」
- 「〇〇は、いわば『□□』のようなものです。」
- 「〇〇に相当します。」
- 「〇〇という言葉が馴染み深いかもしれません。」

## 5. 用語表記の置換ルール

リライトの最後に**全置換**で通せるリスト。

### 長音記号を落とす

| Before | After |
|---|---|
| ユーザー | ユーザ |
| サーバー | サーバ |
| コンピューター | コンピュータ |
| ブラウザー | ブラウザ |
| メモリー | メモリ |
| プロセッサー | プロセッサ |
| プロバイダー | プロバイダ |
| クラスター | クラスタ |
| パラメーター | パラメータ |
| プロパティー | プロパティ |
| ヘッダー | ヘッダ |
| フッター | フッタ |
| ルーター | ルータ |
| インタープリター | インタプリタ |

ただし、慣用上長音を残すもの:
- レイヤー(レイヤ より一般的)
- レイヤード(慣用)
- フィルター(個人差あり、筆者の癖を見て判断)

### 識別子・予約語のバッククォート化

文中に裸で出ている識別子は `xxx` で囲う。

例:
- `key` `tag` `stateNode` `child` `sibling` `return` `index` `alternate`
- `redirect_uri` `client_id` `state` `code_verifier`
- `xmin` `xmax` `t_xmin` `t_xmax`
- `HostRoot` `HostComponent` `FiberRootNode`
- `RFC 6749` 自体はコードにしない(プレーンテキスト)

### 英数字と日本語の間に半角スペース

| Before | After |
|---|---|
| OAuth2.0と | `OAuth 2.0 と` |
| Handle式 | `Handle 式` |
| Reactの | `React の` |
| JWTを | `JWT を` |
| 100msで | `100ms で` |

スペースは**1個**。両側に入れる。

### 概念語の精度上げ(文脈次第)

| 曖昧 | 精度の高い候補 |
|---|---|
| 認証 | 「ID 連携」「ログイン」「本人確認」(用途で選ぶ) |
| 証明書 | 「証明トークン」「認証結果の証明書」 |
| 仮想 DOM | (本論の最初で必ず「本質的には仮想 DOM ではない」と注釈する) |
| スナップショット | 文脈により「トランザクション」と区別 |

## 7. 「橋渡し文」のストック

箇条書き・表・図・コードブロックの**直前**に置く一文。

| 後続要素 | 橋渡し文の候補 |
|---|---|
| 箇条書き | 「以下のような〜があります。」「特徴は以下の通りです。」「以下の通りです。」 |
| 表 | 「整理してみましょう。」「以下のようになります。」「まとめると、以下のようになります。」「以下の表に整理しました。」 |
| mermaid | 「以下のような流れで行われます。」「フローを図示すると、以下のようになります。」 |
| コードブロック | 「実際の PostgreSQL ソースコードでは、以下のように定義されています。」「以下のようなコードになります。」「具体的には、以下のようなコードを記述します。」 |
| 図(画像) | 「以下の図を見てみましょう。」「以下のような構成です。」(画像直前に独立した橋渡し文はないことも多い) |
