## 本論の典型段落

### 例 1: 概念の導入 (sikkari-oauth-oidc/05)

```
**認可コード**は、アクセストークンを引き換えるための一時的な「引換券」です。
特徴は以下の通りです。

- 一度使うと無効になる
- 有効期限は短い方が望ましい（一般的に数分程度）
- 直接アクセストークンを受け取るのではなく
  認可コードを経由することでセキュリティが向上する
```

ポイント:
- 概念名を文頭で太字宣言、その直後に比喩 `「引換券」`
- 「以下の通りです。」で箇条書きに橋渡し
- 箇条書き 3 項目目で**1 項目内改行**

### 例 2: 仕様引用と言い換え (sikkari-oauth-oidc/02)

```
OAuth と OIDC は、**目的が全く異なるユースケース**に対応しています。

一般的に、OAuth は 認可のプロトコル、
OIDC は 認証のプロトコル として表現されることが多いです。

しかし、本書ではより本質的に理解するために、
**OAuth は「リソース認可のプロトコル」**、
**OIDC は「ID 連携のプロトコル」** と表現します。
```

ポイント:
- 一般的見解 → 本書の立場、と段階を踏む
- 「しかし、」で立場を切り替える
- 太字の中に `「〜」` を入れ子で使う

### 例 3: 図/コードの後の補足 (postgres-internal-mvcc-index)

```
このようにして、単一論理行に対する複数の物理タプルのうち、どの物理タプルが可視であるかを判定します。

単一論理行に対する複数の物理タプルが存在した場合でも、可視性チェックアルゴリズムによって
最終的に**単一論理行に対応するタプルが 0 or 1 個の物理タプルに絞り込まれ**ます。
```

ポイント:
- 「このようにして、」で説明を締める

## 太字の振りかけ方

筆者の太字は**1 段落に 1〜3 箇所**、**単語・短句単位**で散らされます。文単位の太字は基本的にしません。

実例:

```
このとき Zenn は内部で Google から
**「このユーザは確かに Google アカウントを持っており、ログインに成功した」**
という証明書を受け取っています。
```
→ 内側のセリフ部分だけ太字、地の文の述語は太字にしない。

```
具体的には、
**アクセストークン**と呼ばれる「リソースにアクセスできる鍵」を利用することで、
リソースへのアクセスが可能になります。
```
→ 概念語のみ太字。

```
最終的に**単一論理行に対応するタプルが 0 or 1 個の物理タプルに絞り込まれ**ます。
```
→ 重要な結論の述語部分を太字。最後の「ます」だけ外す癖がある(これは特徴的)。

```
これは **セキュリティ向上のために必要なステップ** です。
```
→ 補語部分を太字、述語は外。

避けるべき形:
```
**これはセキュリティ向上のために必要なステップです。**
```
(文全体を太字にしない)

## 表の使い方

筆者は**箇条書きより 2 列表を好む**傾向があります。

### パターン 1: プロパティ表 (how-react-works)

```markdown
| プロパティ名 | 説明                                                         |
| ------------ | ------------------------------------------------------------ |
| `key`        | コンポーネントの一意な識別子                                 |
| `tag`        | コンポーネントのタイプ (関数コンポーネント、DOM 要素など)    |
| `stateNode`  | コンポーネントのインスタンス (実際に対応する DOM ノードなど) |
```

ポイント: 識別子はバッククォート、説明にも `()` で補足を入れる。

### パターン 2: 比較表 (sikkari/02)

```markdown
| 項目                       | OAuth                                            | OIDC                                          |
| -------------------------- | ------------------------------------------------ | --------------------------------------------- |
| **目的**                   | リソースに対する認可                             | 認証結果の<br>証明トークンを用いた<br>ID 連携 |
| **主な用途**               | 第三者サービスにリソースへのアクセス権を与える   | 第三者サービスにログインする                  |
```

ポイント:
- 行ヘッダ(左列)が太字
- セル内が長くなる場合は `<br>` で縦に開く

### パターン 3: 役割表 (sikkari/04)

```markdown
| 役割                                   | 名称             | 役割の説明                                                                           |
| -------------------------------------- | ---------------- | ------------------------------------------------------------------------------------ |
| リソースの持ち主                       | リソースオーナー | リソースの持ち主。<br>リソースへのアクセスを許可する主体                             |
```

ポイント: 3 列以上の表でも、説明セルに `<br>` を入れて縦に開く。

表の前には必ず橋渡し文を:
- 「以下のようになります。」
- 「以下の通りです。」
- 「整理してみましょう。」
- 「まとめると、以下のようになります。」

## callout `:::message` の使い分け

### 注意喚起 — `:::message alert`

```markdown
:::message alert
**OAuth を ID 連携・外部ログインに利用しないこと！**

OAuth はリソースを認可するための仕組みであり、ログイン用途に設計されたものではありません。
OAuth をログイン用途に転用すると意図されない使われ方となり、セキュリティ上の問題が発生する可能性があります。

詳細は後の章で解説しますが、ログインが必要な場合は OIDC を使用しましょう。
「餅は餅屋」です。
:::
```

ポイント:
- 1 行目は **太字 + `！`** で警告を強く
- 中身は本論の文体(敬体)
- 締めに諺・成句を置くことがある

### 補足 — `:::message`

```markdown
:::message
コード生成アプローチは watch オプションを提供していることが多く、
watch モードを利用することでコード生成の手間を減らすことができます。
:::
```

ポイント: 本論を脱線せず流したい補足はこれで囲う。

## 脚注 RFC 参照のパターン

```markdown
**認可コード**は、アクセストークンを引き換えるための一時的な「引換券」です。[^rfc6749-auth-code]

[^rfc6749-auth-code]: RFC 6749 Section 1.3.1 において、認可コードは「クライアントが認可サーバーを仲介してリソースオーナーから取得する」と説明されています。https://www.rfc-editor.org/rfc/rfc6749.html#section-1.3.1
```

ポイント:
- 脚注 ID は `^仕様名-内容` (例: `^rfc6749-auth-code`, `^oidc-core-intro`)
- 脚注本文は「〇〇 において、〜と説明されています。」の構文
- URL は本文の最後に裸でつける

## このブックについて

## 本論で「使ってはいけない」表現の実例

筆者の本論にはほぼ出てこない表現:

- 文末の `！` (本論では非常に稀)
- 文末の `・・・` (本論では使わない)
- 文末の `〜` (口語用法。本論では使わない)
- 「だ」「である」常体
- 「〜してくれる」のフランクなニュアンス(避けがち、ただし時々出る)
- 「マジで」「ガチで」「めちゃくちゃ」などの俗語
- 「やばい」「すごい」「最高」などの感情語
- 顔文字、絵文字(emoji frontmatter 以外)

逆に、本論で**よく出る**表現:
- 「〜と言えます」「〜と考えられます」「〜と思います」
- 「〜していきます」「〜していきましょう」
- 「〜することができます」(冗長だが筆者は使う)
- 「以下のような〜」「以下の通りです」
- 「ここで」「では、」「つまり、」「具体的には、」
- 「余談ですが、」
- 「本書では」「本記事では」「今回は」
